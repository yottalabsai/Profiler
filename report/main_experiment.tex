\section{Evaluation}
\label{sec:experiments}

\subsection{Experimental Setup}

\textbf{Hardware.}  We evaluate on two production GPU architectures:
\begin{itemize}[leftmargin=1.5em]
  \item \textbf{NVIDIA RTX PRO 6000 Blackwell Server Edition}: 188 SMs, 96\,GB HBM3e,
    5th-generation Tensor Cores (WMMA BF16/TF32), peak BF16 throughput 1,457\,GB/s memory bandwidth.
  \item \textbf{NVIDIA H100 SXM5 80\,GB}: 132 SMs, 80\,GB HBM3, WGMMA units,
    3,350\,GB/s memory bandwidth.
\end{itemize}

\textbf{Software.}  PyTorch 2.11; CUDA 12.x; Nsight Systems 2025.1; Nsight Compute 2025.4.1;
Python 3.11.

\textbf{Measurement protocol.}  All timing uses \texttt{torch.cuda.Event(enable\_timing=True)}
with 50 warm-up iterations and 200 measured iterations, reporting the median.  Profiling is
performed with 10-iteration \nsys captures and per-unique-kernel \ncu replays.

\textbf{Baselines.}  We compare three configurations:
\begin{enumerate*}[label=(\arabic*)]
  \item \emph{Eager FP32}: standard PyTorch inference without \texttt{torch.compile}, FP32
    precision;
  \item \emph{Inductor FP32}: \texttt{torch.compile(backend="inductor")} with default settings,
    FP32 precision (\TODO{fill in measured values});
  \item \emph{PGO BF16}: \sys PGO backend with BF16 casting and all applicable FX passes.
\end{enumerate*}

\subsection{MLP Activations}

The MLP workload highlights how \sys diagnoses activation-function bottlenecks that are invisible
to timing-only profilers.  The baseline profile reveals \texttt{ipc\_active = 0.05} on the
\texttt{tanh} activation kernel, caused by SFU pipeline serialization: the \texttt{\_\_tanhf}
special function routes through a 4-cycle-per-instruction SFU pipeline, stalling all other warps.
BF16 casting activates Tensor Core paths on all four linear layers (TC utilization $0\% \to 39\%$
on the fused \texttt{addmm}+\texttt{relu} Triton kernel), and the activation normalization pass
replaces \texttt{tanh} with \texttt{gelu(approximate='tanh')} in the FFN context, eliminating the
SFU stall.  Total speedup: \textbf{6.2$\times$} (355,500\,ns $\to$ 57,000\,ns).

\subsection{SDPA Attention}

The SDPA workload demonstrates spill elimination.  The baseline \texttt{fmha\_cutlassF\_f32}
kernel uses 168 registers/thread, generating 757,760 local-memory spill events per call.  Despite
58.7\% Tensor Core activity (the kernel does use TCs), achieved occupancy is capped at 14.1\% and
SM throughput at 51.4\% by the spill overhead.  After BF16 casting, Inductor selects a lower
register-pressure attention kernel; spills drop to zero, occupancy rises to ${\sim}27\%$, and the
attention kernel alone contributes ${\sim}3.4\times$ speedup.  Combined with the mm kernels
(2.5$\times$) and LayerNorm (1.3$\times$), total speedup reaches \textbf{2.7$\times$}
(360.7\,$\mu$s $\to$ ${\sim}$132\,$\mu$s).

\subsection{ConvBlock}

The ConvBlock workload illustrates a non-GEMM bottleneck pattern.  Two bottlenecks dominate the
baseline profile: (1) \texttt{convertTensor\_kernel} launches (60 total, 10.5\% of wall time)
caused by cuDNN's NHWC-to-NCHW coercion for FP32 inputs, eliminated entirely by FP16 casting; and
(2) a 7-kernel Triton decomposition of BatchNorm at inference time, where two channel-mean
reduction passes recompute statistics that are frozen constants, addressed by constant-folding the
BatchNorm into a single scale-bias fuse.  Kernel time speedup: \textbf{2.2$\times$}; wall-time
speedup with CUDA Graphs: \textbf{3.5$\times$}.

\subsection{LSTM Sequence Encoder}

The LSTM workload reveals a distinct attribution and optimization pattern.  Under
PyTorch~2.11 + \texttt{torch.compile(backend="inductor")} on H100~PCIe,
\texttt{nn.LSTM} dispatches to \texttt{aten::\_cudnn\_rnn}, a single opaque cuDNN operator that
executes as \texttt{Kernel2}---cuDNN's internal FP32 SIMT path for LSTM gate computation.
Dynamo produces \emph{zero} traceable FX subgraphs for \texttt{nn.LSTM}: the operator enters
via \texttt{torch.\_VF.lstm}, a C extension entry point that Dynamo cannot graph-capture.
All FX graph passes are therefore inapplicable; the PGO backend is bypassed entirely.

The profiler correctly attributes timing to \texttt{aten::\_cudnn\_rnn} (98.7\% of total
kernel runtime) but reports null hardware counters: cuDNN's internal kernels are opaque to
\ncu at the operator-aggregation level.  Attribution is \texttt{UNATTRIBUTED} for 85\% of
kernel runtime, the highest rate across all evaluated workloads.

Bottleneck diagnosis proceeds by \emph{kernel-name inference}: \texttt{Kernel2} is the
cuDNN FP32 SIMT path; on H100, BF16 inputs route cuDNN to
\texttt{sm90\_xmma\_gemm\_bf16bf16\_*} (WGMMA Tensor Core path).
A single \texttt{model.to(bfloat16)} call achieves \textbf{1.77$\times$} total speedup
(27.20\,ms $\to$ 15.39\,ms); \texttt{aten::addmm} (the linear classifier head) gains
2.41$\times$ via the BF16 cuBLAS path, and temporal mean-pooling gains 1.32$\times$ from
smaller BF16 operands.  This workload demonstrates that \sys's closed-loop protocol applies
even when hardware counters are unavailable: the profiler identifies the cuDNN bottleneck
class, kernel-name inference suggests the fix, and re-profiling confirms the gain.
